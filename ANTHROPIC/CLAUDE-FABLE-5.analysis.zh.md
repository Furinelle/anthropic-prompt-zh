# Claude Fable 5 System Prompt 分析

> 本文基于仓库中的 `CLAUDE-FABLE-5.md` 与中文译文进行提示词工程分析。该材料属于非官方归档，不能视为 Anthropic 官方文档或真实线上行为的证明。

## 总体判断

Claude Fable 5 的 system prompt 不是单纯的“角色设定”，而是一份面向产品级对话智能体的运行规约。它同时承担了身份说明、产品知识边界、安全策略、搜索策略、工具调用协议、文件处理规范、引用规范、MCP App 选择流程、Artifacts 能力说明和运行环境配置等职责。

从提示词工程角度看，它更像一套“运行时宪章 + 工具路由手册 + 安全策略包”。它的核心设计目标不是让模型显得更聪明，而是让模型在复杂产品环境中稳定、可控、可审计地工作。

## 结构分层

这份提示词大致可以拆成九层。

1. **基础行为层**

   `claude_behavior` 规定 Claude 的产品身份、拒绝策略、儿童安全、法律/财务建议、语气格式、用户福祉、政治与伦理议题处理，以及知识截止日期。它是整份提示词的行为基线。

2. **记忆与状态层**

   `memory_system` 说明当前用户是否启用记忆；`persistent_storage_for_artifacts` 则定义 Artifacts 可使用的持久化键值存储。这一层把“对话记忆”和“Artifact 内部状态”区分开，避免把两种状态混为一谈。

3. **MCP App 选择层**

   `mcp_app_suggestions` 规定什么时候搜索连接器、什么时候建议用户启用连接器、什么时候可以直接调用已选连接器。它的重点是用户选择权，而不是工具可用就直接调用。

4. **计算机使用层**

   `computer_use` 规定文件生成、Skill 读取、上传文件处理、输出目录、Artifact 使用条件和包管理约束。这一层把“能不能回答”转化为“是否应该创建真实文件、放到哪里、如何交付”。

5. **搜索与检索层**

   `search_instructions` 详细定义了何时搜索、搜索多少次、如何选择工具、如何处理当前性信息、如何遵守版权限制。它显著降低了模型凭旧知识回答动态问题的风险。

6. **图像搜索层**

   `using_image_search_tool` 规定图像搜索的触发条件、禁用类别和展示方式。它把图像从“装饰性输出”限定为“能增加理解价值时才使用的补充信息”。

7. **工具定义层**

   `Tool Definitions` 直接嵌入工具描述和 JSON schema。这里的提示词不只是自然语言指令，还包含工具契约，模型需要按这些契约构造调用。

8. **Artifacts 中的 API 能力层**

   `anthropic_api_in_artifacts` 描述 Artifacts 可以调用 Anthropic API，并说明如何处理 `data.content`、结构化输出、网页搜索、文件输入和上下文管理。这一层让 Artifact 从静态文件变成可嵌入 AI 能力的小应用。

9. **引用与运行环境层**

   `citation_instructions`、`available_skills`、`network_configuration`、`filesystem_configuration` 负责把搜索结果引用、可用 Skill、网络白名单和文件系统边界讲清楚。这些内容更接近运行环境说明，而不是人格设定。

## 关键工程模式

### 保留结构标签

原文使用 `claude_behavior`、`search_usage_guidelines`、`artifact_usage_criteria` 这类结构标签，而不是普通文章标题。这说明它很可能服务于内部配置、生成、拼装或审计流程。中文译文保留这些标签并补充中文括注，是更专业的处理方式，因为它同时保留了原始结构和中文可读性。

### 用显式触发条件减少自由发挥

提示词大量使用“何时使用”“何时不使用”“必须”“不要”“例外”等形式。例如搜索策略会区分稳定知识、快速变化信息、当前职位、未知实体、单一事实查询和复杂研究查询。这种写法本质上是在构建一个轻量级路由器，让模型先分类任务，再决定行动。

### 安全策略不是一个模块，而是贯穿全局

儿童安全、武器、有害内容、心理健康、版权、法律财务建议、图像搜索限制等并不只出现在一个总安全章节中，而是在不同能力章节重复出现。这种冗余不是坏事：当模型进入搜索、图像、文件、Artifact 或工具调用场景时，相关边界会再次被提醒。

### 产品知识采用“查证优先”

提示词明确要求在被问及 Anthropic 产品或功能时，优先搜索官方文档，而不是依赖内置知识。这是一个很成熟的产品级设计，因为模型对自身产品的知识最容易过时，也最容易让用户误以为是官方承诺。

### 工具使用强调用户授权

MCP App 部分的重点不是“能调用就调用”，而是“在合适时机向用户展示选择”。尤其对第三方消费类 App，即使已连接也需要通过选择器让用户确认。这体现了产品 UX 与权限边界：工具调用不仅是技术问题，也是用户代理权问题。

### 文件生成被当成真实交付

`computer_use` 强调用户要求创建文件时必须实际创建文件，并放入输出目录再呈现。这比“把内容贴在聊天里”更接近生产环境的交付逻辑。它还要求在写文件前读取相关 Skill，说明 Anthropic 把文件生成看作多格式、多约束的专业任务。

### 版权策略极其保守

搜索部分的版权规则非常细，甚至对引用长度、引用次数、歌词和文章结构复述都做了硬性限制。这说明搜索增强型模型不仅需要回答正确，还必须避免把检索结果变成版权替代品。对提示词工程来说，这是一个典型的“能力越强，边界越细”的例子。

## 行为倾向

这份提示词会把 Claude 推向以下行为风格：

- **更少凭记忆回答当前问题**：凡是当前职位、政策、价格、产品版本、近期新闻或未知实体，都倾向搜索。
- **更强的拒绝边界**：涉及未成年人性化、武器制造、恶意代码、危险自残细节和版权复制时，拒绝会更坚定。
- **更低的格式侵入性**：语气与格式部分要求少用标题、列表和粗体，除非任务确实需要。
- **更注重用户自主权**：法律、财务、MCP App 选择、心理健康和用户结束对话等场景，都强调不要替用户作过度决定。
- **更像产品内代理而不是裸模型**：它会搜索、读文件、创建文件、调用工具、推荐连接器、使用 Artifacts 和处理 API 响应。

## 对提示词工程的启发

1. **不要把长 prompt 写成一篇文章**

   Fable 5 的提示词更像配置文档：有结构标签、模块边界、触发条件、例外、示例和工具 schema。复杂系统提示词应当可导航、可审计、可局部替换。

2. **把“何时行动”写清楚**

   很多提示词只告诉模型“要做什么”，但产品级提示词还要告诉模型“什么时候做、什么时候不做、做之前检查什么”。搜索、图像、MCP、文件创建都是这样。

3. **为动态知识建立查证路径**

   如果信息可能变化，不应试图用更强语气让模型“记住”，而应指定可靠来源、搜索时机和回答方式。Fable 5 对 Anthropic 产品知识的处理就是这种模式。

4. **把安全规则嵌入能力场景**

   单独写一段安全政策不够。只要某个能力会扩大风险面，例如搜索、图像、文件、API、外部 App，就应该在对应场景内写清边界。

5. **工具 schema 是 prompt 的一部分**

   工具描述、参数名、枚举值、返回值和工作流说明会直接影响模型如何调用工具。把工具契约写清楚，比单纯告诉模型“你可以使用工具”有效得多。

6. **示例要包含正例和反例**

   提示词中大量使用示例、反例和“不要这样做”。这能减少模型在边界模糊处自由解释规则的空间。

## 值得注意的风险

1. **提示词很长，可能产生局部冲突**

   长 system prompt 的风险是不同章节之间出现优先级不清或重复表达。例如“自然散文、少用列表”和“复杂研究需要结构化综合”之间就需要模型自行平衡。

2. **硬编码日期会快速过期**

   提示词中出现具体当前日期和知识截止日期，这有助于当时的行为校准，但也意味着材料一旦脱离原运行环境，就容易过时。

3. **产品声明不可直接当真**

   其中关于 Claude Fable 5、Claude Mythos 5、Claude Cowork 等产品的描述，作为提示词材料有分析价值，但不应视为真实产品路线图或官方事实。

4. **工具能力强，权限边界必须清楚**

   MCP、文件系统、浏览、Artifacts API 都扩大了模型的行动范围。提示词用大量规则约束这些能力，说明 agentic 产品真正困难的地方不是“接入工具”，而是“接入后如何不越界”。

5. **版权策略可能牺牲部分用户体验**

   版权规则非常保守，能降低法律与平台风险，但在某些用户只需要短摘录、对比或文本分析时，也可能导致模型显得过度谨慎。

## 中文译文维护建议

- 结构标题继续采用“原文标签（中文释义）”，不要只翻译成中文。
- Claude、Anthropic、API、MCP、Artifacts、Skills、模型名、工具名和 schema 字段继续保留英文。
- Markdown 正文中的说明性文字应尽量译成自然中文；代码、JSON schema、枚举值和模型字符串保持原样。
- 对技术表达保持中英间空格，例如 `API 调用`、`JSON 格式`、`MCP 服务器`、`Artifacts 中`。
- 对明显来自产品界面的英文功能名，可以保留原文；如果会影响理解，再补中文解释。

## 简短结论

Claude Fable 5 的提示词展示了一种成熟的产品级 system prompt 形态：它不是靠一个人格设定驱动模型，而是用分层规则、工具契约、触发条件、安全边界和示例来塑造可控行为。对提示词工程学习者来说，最值得借鉴的不是某一句具体措辞，而是它把复杂智能体拆成多个可审计模块的方式。
